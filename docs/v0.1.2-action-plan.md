- Two frictions to patch before release:

  2. Tool responses don't expose effective mode — downstream LLM mislabeled a live response as mock from data shape alone.

## Stream A — `sample_messages` timestamps

**Session**: Claude Code #1
**Branch**: `git checkout -b fix/sample-msg-timestamps`
**Agent invoked**: `ros2-adapter-engineer`
**Skills referenced**: `write-pure-parser`

Open a fresh Claude Code session in the repo, then paste:

```
Use the ros2-adapter-engineer agent.

Bug: TopicForge's sample_messages tool returns timestamp=0 in live mode because
the adapter shells out to `ros2 topic echo --once`, which doesn't emit receive
times. Confirmed today against ROS2 Jazzy.

Goal: return real receive timestamps from the live adapter.

Approach: switch the CLI invocation to `ros2 topic echo --csv --once`. The CSV
output includes timestamp_sec and timestamp_nanosec columns. If --csv proves
unreliable for some message types (nested structs, arrays), fall back to
`--full-length --truncate-length 0` and extract header.stamp from the YAML body
when present.

Constraints (per CLAUDE.md):
- Pure parser is a module-level function named parse_csv_echo (or similar),
  living in src/topicforge/adapters/ros2_live/ next to the existing parse_*
  functions. Follow the write-pure-parser skill.
- The parser must be unit-testable without ROS2 — use captured CLI output as
  fixtures under tests/fixtures/.
- Adapter errors raise AdapterError with a user-safe message. No silent fallback
  to timestamp=0 — if parsing fails for a row that should have a timestamp,
  raise.
- Don't touch mock mode behavior.
- Tool handler in src/topicforge/tools/ stays thin; all logic lives in the
  adapter or parser.

Out of scope for this branch: do NOT add a mode_effective field. That's
happening on a parallel branch (feat/mode-effective). If you see a merge
conflict later, it's expected.

Procedure:
1. Read src/topicforge/adapters/ros2_live/ in full (subprocess wrapper + parsers).
2. Read .claude/skills/topicforge/write-pure-parser/SKILL.md.
3. Capture a sample of `ros2 topic echo --csv --once /chatter` output if a ROS2
   talker is available; otherwise document the expected CSV shape from the
   ros2cli source.
4. Propose a short execution plan before editing.
5. Implement: parser + subprocess invocation update + unit tests.
6. Run `make check`, report.
7. Commit on this branch with a clear message and push.

When done, give me the branch name and a 3-line summary of what changed so I
can review and merge.
```

## Stream B — `mode_effective` everywhere

**Session**: Claude Code #2
**Branch**: `git checkout -b feat/mode-effective`
**Agent invoked**: main session (cross-layer change, no sub-agent)

```
Add a mode_effective field to every TopicForge tool response so downstream LLMs
can't confuse live and mock output.

Background: a Claude Code session running TopicForge in live mode hallucinated
"mock mode — no live ROS2 detected" because the response payload didn't carry
the effective mode. health_check exposes it, but list_topics, get_topic_info,
sample_messages, and analyze_bag don't.

Goal: add `mode_effective: Literal["live", "mock"]` to ListTopicsResult,
TopicInfo (or its wrapper result), SampleMessagesResult, and BagAnalysis in
src/topicforge/models/schemas.py. Each tool's response carries the mode the
adapter actually ran in.

Constraints (per CLAUDE.md):
- Pydantic models are frozen with extra="forbid" via the shared _CONFIG.
  Add the field to the schema definitions directly — this is the cleanest path.
  Services then populate it at result construction time.
- The adapter knows its effective mode. Expose it via a new property on the
  RosAdapter protocol in adapters/base.py:
      @property
      def effective_mode(self) -> Literal["live", "mock"]: ...
  Implement it on Ros2CliAdapter (returns "live") and MockAdapter
  (returns "mock").
- Services read adapter.effective_mode and pass it into the result construction.
- Tool handlers stay thin — they receive the fully-populated model from the
  service and return it.
- Update existing tests to assert mode_effective is present and correct for
  both live and mock paths.

Out of scope for this branch: do NOT change the CSV echo / timestamp parsing
logic. That's happening on a parallel branch (fix/sample-msg-timestamps). If
you see a merge conflict later in adapters/ros2_live/, it's expected — keep
your changes there minimal (only the effective_mode property).

Procedure:
1. Read in order: src/topicforge/models/schemas.py,
   src/topicforge/adapters/base.py,
   src/topicforge/adapters/mock/,
   src/topicforge/adapters/ros2_live/,
   src/topicforge/services/.
2. Propose a short execution plan.
3. Implement: schema → adapter protocol → adapter implementations →
   services → tests.
4. Run `make check`.
5. Commit on this branch with a clear message and push.

When done, give me the branch name and a 3-line summary so I can review
and merge.
```

## Stream C — MCP 02 spec draft

**Session**: Claude Code #3
**Branch**: `git checkout -b docs/mcp-02-spec`
**Skills invoked**: none (the prompt below imposes the full structure; no external skill is required)

> **Note on skill references.** Earlier drafts of this plan referenced `product-management:product-brainstorming` and `product-management:write-spec`. Those skills are not installed in this environment and are not on the install path for v0.1.2. The prompt below is self-contained — it carries the brainstorming framework and the spec template inline.

```
I want to draft the spec for MCP 02 of the robotics/CV pack — the next MCP
after TopicForge. TopicForge v0.1.2 is shipping in parallel with this work;
no dependency on the code work.

You do not need any external skill for this work. The structure is imposed
directly in this prompt. Read docs/product-plan.md, CLAUDE.md, and docs/pro.md
in full first to inherit TopicForge's conventions, then follow the two
phases below.

PHASE 1 — Pick MCP 02.

Candidates listed in CLAUDE.md section 1 and docs/product-plan.md section 4:
  - Vision Dataset Inspector
  - Synthetic Data Pipeline Controller
  - URDF Generator / Validator
  - Robotics Trajectory Visualizer

Rank them against these five criteria:
1. Persona proximity to TopicForge: same ROS2 / robotics / CV engineer who'd
   install v0.1.2. We want pack synergy.
2. Code overlap: minimal. The pack should feel like complementary tools, not
   five variations of the same MCP.
3. AI agent leverage: the "an LLM with this tool makes the user 10x faster"
   story should be obvious.
4. MVP shippability: same shape as TopicForge — 5 read-only MCP tools, mock +
   live modes, an adapter abstraction equivalent to RosAdapter — inside 2
   weeks of part-time work.
5. Defensibility: nobody else has packaged this for the MCP world yet.

Output a short ranking with reasoning, recommend a winner, surface the
strongest counter-argument.

PHASE 2 — Draft the spec.

Draft a one-pager at docs/projet-file/mcp-02-spec.md. Follow the same section
structure as TopicForge's own product-plan.md and CLAUDE.md:
  - 1. Identity (one-liner, target users)
  - 2. Locked MVP scope (5 tools max, read-only, table format)
  - 3. Architecture sketch (layer diagram, adapter protocol if applicable)
  - 4. Runtime modes (mock / live / auto if applicable)
  - 5. Stack (Python 3.11+, mcp >= 1.0, pydantic >= 2.6, pytest, ruff, hatchling)
  - 6. Engineering principles (inherit from TopicForge — clean architecture,
       graceful degradation, type hints, structured outputs)
  - 7. Phase 1 targets
  - 8. Risk register
  - 9. Monetization (free MVP, pro tier later)

Keep it tight — one page in printed form, max ~250 lines. Reuse TopicForge
conventions verbatim where they apply; flag only the genuine differences.

When done, commit on this branch and push. Give me the recommended MCP 02 +
a 5-bullet TL;DR.
```

## Sync points & merge gates

```
Day 0 (today)
├── Start Stream A   ┐
├── Start Stream B   ├── all three in parallel, separate sessions
└── Start Stream C   ┘

Day 0 +1h
├── Review + merge B → main (foundational schema)
├── Rebase A on main if needed, review + merge A → main
├── Review C output — merge C → main when you're happy with the spec
└── Start Stream D off main (D1 QA)

Day 0 +2h
├── D1 PASS → D2 docs sync → commit
├── D2 done → D3 release-checklist
└── v0.1.2 live on PyPI
```

**Gates that must hold before D starts:**
- A merged. B merged. CI green on main.
- `pyproject.toml` is still at 0.1.1 (D2 bumps it).
- No uncommitted changes in `docs/projet-file/` outside what C produced (so C doesn't get sucked into D).

**If a stream blocks:**
- A blocked on parser quirks → drop the `--csv` fallback to YAML extraction and ship partial coverage. Note as known limitation in CHANGELOG.
- B blocked on schema conflict with A → land B first, A rebases.
- C blocked on candidate ambiguity → don't force a winner; commit a "comparison matrix" doc instead, decide in next session.
- D1 FAIL → fix on `release/v0.1.2` branch, re-run D1. Do not skip.
