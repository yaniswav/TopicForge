After several years on DDS-heavy stacks at Thales, one pattern became obvious: the moment you let an LLM agent touch the control bus — even with good intentions — you have a liability conversation you cannot win. The safety review alone kills most proposals. The default open-source answer today (rosbridge-based MCP servers with a full write path) is exactly that shape.

TopicForge takes a different position. It is read-only by architecture, not by configuration. There is no write path to misconfigure, no permission flag you could accidentally flip, no command path back to the robot. The MCP client can see your ROS2 graph — topics, message types, bag recordings, live samples with real publish-time timestamps — but it cannot touch it. That is the sentence that ends the security review.

v0.3.0 ships multi-vendor DDS observability: two OSS Python adapters — Eclipse CycloneDDS and eProsima Fast DDS — both joining the bus as read-only DDS-RTPS participants. By the OMG-DDS-RTPS protocol guarantee, both adapters observe every conformant vendor on the wire: RTI Connext, OpenDDS, CoreDX, even Rust implementations like Dust DDS. Language and vendor are irrelevant at the wire level. The OMG May 2025 interop matrix validates 47/47 pairs across six implementations, one in Rust — receipts at https://omg-dds.github.io/dds-rtps/test_results.html

Three MCP tools (participants, QoS-mismatch diagnostics, raw DDS samples) extend the same read-only-by-architecture contract that v0.1.x established for the ROS2 graph. The MiddlewareAdapter protocol does not expose a write method — one umbrella, two backends, one safety review.
